# Data Dictionary

**Project:** Federal Spending Anomaly Detection
**Author:** Hari Acharya
**Layer documented:** Curated star schema in the `fed_spending` PostgreSQL database, plus its direct source table.

## Purpose

This document describes every table and column in the curated layer of the project, along with its data type, meaning, source, and any transformation applied. It is the reference that lets anyone trace a value in a dashboard back to its origin in the source data.

A note on honesty about the data. The data is real and public, pulled from the USASpending API. The stakeholders and the review team in this project are simulated for portfolio purposes. The dataset is NASA prime contract awards (award type codes A, B, C, D) that were active during fiscal years 2022 and 2023, meaning October 2021 through September 2023. It is described as awards active during the window, not awards that started then, because the API matches awards that had activity in the window.

## Data lineage

The data moves through four stages. Each stage is a separate, named object so ingestion stays separate from cleaning and from modeling.

1. `data/awards_raw.csv` is the raw pull from the API. It is not committed to Git.
2. `stg_awards_raw` is the raw CSV loaded into PostgreSQL with every column as text and column names tidied to snake_case. No transformations.
3. `stg_awards_clean` is the curated-ready table. NAICS and PSC bundles are parsed into code and description columns, recipient names are standardized, the amount is converted to a number, and the dates are converted to real dates.
4. The star schema (`dim_recipient`, `dim_naics`, `dim_psc`, `dim_date`, `fact_award`) is the dimensional model the dashboards and rules read from.

There is deliberately no agency dimension. Every award in the dataset is NASA, so an agency table would have a single row and add nothing.

## Source table: stg_awards_clean

This is the direct source for the star schema. It holds 16,866 rows, one per unique award.

| Column | Type | Description | Source | Transformation |
|---|---|---|---|---|
| generated_internal_id | text | Permanent award identifier from the API | API field generated_internal_id | Used as the duplicate key during ingest |
| award_id | text | Human readable award identifier (PIID) | API field Award ID | None |
| recipient_name | text | Vendor name | API field Recipient Name | Trimmed, repeated spaces collapsed, uppercased, kept gentle on purpose so different vendors are not wrongly merged |
| award_amount | double precision | Award dollar amount | API field Award Amount | Converted from text to a number |
| start_date | date | Period of performance start date | API field Period of Performance Start Date | Converted from text to a real date |
| end_date | date | Period of performance end date | API field Period of Performance End Date | Converted from text to a real date |
| naics_code | text | NAICS industry code | Parsed from the API NAICS bundle | Split out of bundle text like {'code': '541712', 'description': '...'} |
| naics_description | text | NAICS industry description | Parsed from the API NAICS bundle | Split out of the same bundle |
| psc_code | text | Product or Service Code | Parsed from the API PSC bundle | Split out of bundle text |
| psc_description | text | Product or Service Code description | Parsed from the API PSC bundle | Split out of the same bundle |

## Curated layer

### dim_recipient

One row per distinct vendor. 4,064 rows.

| Column | Type | Description | Source | Transformation |
|---|---|---|---|---|
| recipient_key | integer (serial) | Surrogate primary key | Generated by the database | Derived |
| recipient_name | text | Standardized vendor name, unique | stg_awards_clean.recipient_name | Deduplicated to one row per distinct name |

### dim_naics

One row per NAICS code. 412 rows.

| Column | Type | Description | Source | Transformation |
|---|---|---|---|---|
| naics_key | integer (serial) | Surrogate primary key | Generated by the database | Derived |
| naics_code | text | NAICS industry code, unique | stg_awards_clean.naics_code | Deduplicated to one row per code |
| naics_description | text | Industry description | stg_awards_clean.naics_description | One description kept per code |

### dim_psc

One row per Product or Service Code. 779 rows.

| Column | Type | Description | Source | Transformation |
|---|---|---|---|---|
| psc_key | integer (serial) | Surrogate primary key | Generated by the database | Derived |
| psc_code | text | Product or Service Code, unique | stg_awards_clean.psc_code | Deduplicated to one row per code |
| psc_description | text | Code description | stg_awards_clean.psc_description | One description kept per code |

### dim_date

One row per calendar day seen in either date column. 3,703 rows. The span reaches back well before the fiscal window because award start dates can be years earlier, which fits the awards active during the window framing.

| Column | Type | Description | Source | Transformation |
|---|---|---|---|---|
| date_key | integer | Primary key, the date written as YYYYMMDD, for example 20221001 | Derived from full_date | Derived |
| full_date | date | The calendar date, unique | Distinct start_date and end_date values | Derived |
| year | integer | Calendar year | Derived from full_date | Derived |
| quarter | integer | Calendar quarter, 1 to 4 | Derived from full_date | Derived |
| month | integer | Month, 1 to 12 | Derived from full_date | Derived |
| fiscal_year | integer | Federal fiscal year, which starts on 1 October | Derived from full_date | A date in October, November, or December counts as the next fiscal year |
| fiscal_quarter | integer | Federal fiscal quarter, 1 to 4 | Derived from full_date | Oct to Dec is 1, Jan to Mar is 2, Apr to Jun is 3, Jul to Sep is 4 |

### fact_award

One row per unique award. 16,866 rows. Grain is one award.

| Column | Type | Description | Source | Transformation |
|---|---|---|---|---|
| award_sk | integer (serial) | Surrogate primary key | Generated by the database | Derived |
| generated_internal_id | text | Permanent award identifier, the natural key, unique | stg_awards_clean.generated_internal_id | None |
| award_id | text | Human readable award identifier | stg_awards_clean.award_id | None |
| recipient_key | integer | Foreign key to dim_recipient | Join on recipient_name | Derived |
| naics_key | integer | Foreign key to dim_naics | Join on naics_code | Empty for the 7 awards with no NAICS code |
| psc_key | integer | Foreign key to dim_psc | Join on psc_code | Derived |
| start_date_key | integer | Foreign key to dim_date for the start date | Join on start_date | Empty for the 1 award with an unparsed start date |
| end_date_key | integer | Foreign key to dim_date for the end date | Join on end_date | Derived |
| award_amount | numeric(18,2) | The award dollar amount, the fact measure | stg_awards_clean.award_amount | Cast from double precision to numeric(18,2) |
| risk_score | numeric(10,2) | Weighted anomaly score | Derived in Phase 4 | Currently 0 for every row, filled by the rule engine |

## Data quality notes

These are recorded honestly rather than hidden or patched, because they affect how the numbers should be read.

- **7 awards have no NAICS code.** Their naics_key is left empty. This is normal for federal data. We document it rather than invent a code.
- **1 award has an unparsed start date.** Its start_date_key is left empty. This is the same single record flagged during the cleaning step, so the count is consistent end to end.
- **Award amounts are heavily right skewed.** The minimum is about negative 2,024,625, which represents de-obligations, money returned rather than spent. The maximum is about 22.4 billion, which is most likely a large indefinite-delivery contract ceiling, not money actually spent. The median is about 120,031 and the average is about 13.2 million. 258 awards have a negative or zero amount. The anomaly rules in Phase 4 need to account for this skew and for de-obligations.
- **The original Award Type column was empty for contracts and was dropped.** The dataset was already filtered to contract types A, B, C, D during ingest, so a separate award type column is not needed in the model.
- **Recipient name standardization was kept gentle on purpose.** Names were trimmed, had repeated spaces collapsed, and were uppercased, but no aggressive matching was done, to avoid merging two genuinely different vendors into one.
